# pjangler — Architecture

> Generated by BMAD `document-project` (deep scan) · 2026-07-13
> Source of truth is the code under `src/`. This document is a navigation aid for AI-assisted development, not a spec.

## Executive summary

**pjangler** (`@delorenj/pjangler`) is a TypeScript **CLI + MCP server** that bootstraps and maintains **33GOD / DeLoNET** projects. It has one job with three faces:

1. **Bootstrap** a new project — register it centrally, render the `CommonProject` scaffold (via `copier`), write a repo-local `.project.json`, and optionally provision a Hermes PM agent.
2. **Audit & migrate** an existing repo against the deterministic "33god project standard" — a set of idempotent parity rules.
3. **Scaffold subsystems** into any repo (`mise`, `docker`, `node`, `hermes-agent`, `agent-hooks`) via composable recipes.

The same core logic is exposed through **two interfaces that share one implementation**: a human-facing `commander` CLI (`src/index.ts`) and a stdio **MCP server** (`src/mcp-server.ts`) for agents. Neither interface holds business logic — both delegate to the same functions (`planProjectInit`, `runAudit`, `runMigration`, `createRecipe`).

**Architecture pattern:** layered CLI with a **command/recipe composition core**, a **plan/apply execution model**, and a **rule-based idempotent reconciler** (the parity engine). It is a stateless tool: the only persistent state it owns is the central project registry (`~/.config/pjangler/projects.yaml`); everything else is derived from, or written into, the target repo.

## Core building blocks

### 1. Command / Recipe composition (`src/commands/`, `src/recipes/`, `src/utils/registry.ts`)

The scaffolding engine is a small composition pattern:

- **`Command`** (`src/commands/Command.ts`) — abstract base. One method, `invoke(): Promise<InvokeResult>`. Provides `writeFile`/`createDirectory`/`fileExists` helpers that are **dry-run-aware** (no-op writes when `context.dryRun`) and a `formatMessage()` that prefixes `[DRY RUN]`. A Command is an atomic file/dir operation (e.g. `AddDockerfile`, `AddMiseToml`).
- **`Recipe`** (`src/recipes/Recipe.ts`) — abstract base. Composes Commands as **ingredients** via `addIngredient(CommandClass)`; `execute()` runs each ingredient's `invoke()` in order, prints an indented banner, then `printNextSteps()`. A Recipe bootstraps a whole subsystem.
- **Registry** (`src/utils/registry.ts`) — two maps, `RECIPE_REGISTRY` and `COMMAND_REGISTRY`, plus `createRecipe(name, ctx)`. This is the single lookup surface used by the CLI (`add`, `recipe run`), the MCP server (`pjangler_run_recipe`), and metadata commands (`recipe list/describe`, `command list/describe`).

Adding a subsystem = write Command classes, write a Recipe that chains them, register both in `registry.ts`. See the `pjangler-dev` / `project-jangler` skill for the authoring workflow.

**Registered recipes:** `mise`, `docker`, `node`, `hermes-agent`, `agent-hooks` (see `component-inventory.md`).

The `hermes-agent` recipe **overrides `execute()`** to suppress the base per-command logging (its commands render their own `@clack/prompts` UI) and to **short-circuit on hard failure** so a cancelled prompt doesn't leave a half-provisioned agent.

### 2. Project registry + bootstrap (`src/project/index.ts`)

Owns project identity and the full bootstrap plan.

- **Central registry:** `~/.config/pjangler/projects.yaml` (override with `PJ_PROJECT_REGISTRY`), `schema_version: 1`. `loadProjectRegistry` / `saveProjectRegistry` (atomic temp-file rename) / `validateProjectRegistry` enforce **unique slug, repo_path, and ticket identifier** across all projects.
- **Schemas:** `ProjectRecord` (registry entry) and `ProjectManifest` (the repo-local `.project.json` projection). `projectManifestFromRegistryProject()` projects one to the other; agent keys are namespaced `<slug>-<role>`.
- **Plan/apply model:** `planProjectInit(input)` returns a `ProjectInitPlan` — a list of **typed actions** with no side effects:
  - `registry.upsert` — add/update the registry entry
  - `copier.copy.commonproject` — the exact `copier copy --trust … --defaults --data k=v …` argv (built by `buildCommonProjectCopierAction`)
  - `project.write-manifest` — write `.project.json`
  - `ticket-provider.create-or-link` — gated behind `--live` (network/cloud)
  - `hermes.provision-agent` — gated behind `--provision-agent`
  `executeProjectInitPlan(plan)` applies the actions, spawning `copier` and writing files **only when `plan.apply`**, and returns changed files + logs + errors. **Dry-run is the default everywhere.**
- **`.project.json` `ticket_provider` block** is the single source of truth for the board binding. `buildTicketProviderBlock()` stores only stable identity (`type`, `workspace`, `identifier`, `board_id`, `state`); **board URLs are derived at runtime**, never persisted (`board_url` is a deprecated no-op input). Supported providers: `plane`, `trello`.
- **`doctorProjectRegistry()`** validates that each project's `repo_path` exists, is a directory, has a `.project.json`, and that source artifacts resolve.
- **Agent-hooks layer decision:** `resolveAgentHooksLayer()` — explicit input wins; then `PJ_AGENT_HOOKS_LAYER` (0/1); otherwise the layer is **skipped when a global `~/.agents/hooks` install exists**, so a fresh project never re-injects hooks into the caller's shared per-user CLI configs. (Mirrored inside the parity engine for generated `mise.toml`.)

### 3. Parity engine (`src/parity/index.ts`)

The largest and most involved module (~2,100 LOC). A deterministic **audit → migrate** reconciler that keeps any repo in parity with the 33god standard.

- **Rule shape:** `interface Rule { id; title; audit(ctx) → AuditFinding; migrate(ctx, finding) → MigrationRuleResult }`. All rules live in one `const RULES: Rule[]` array.
- **Audit** (`runAudit(repo?)`) maps every rule's `audit` over a `Context` (`repoRoot`, `dryRun`, `pjanglerRoot`, `homeDir`) → `AuditReport { ok, rules: AuditFinding[] }`. Each finding has `status` (`pass | fail | warn | skip`), a summary, granular `details[]`, and a `fixable` flag.
- **Migrate** (`runMigration` / `runMigrationForRules`) runs the `migrate` of selected rules → `MigrationReport`. Migration results are `applied | noop | blocked | skipped` and are **idempotent** — re-running yields `noop`. `--dry-run` reports changes without writing; `blocked` signals a condition a human must resolve (e.g. a non-symlink file where a symlink is expected).
- **Formatting:** `formatAuditReport` / `formatMigrationReport` render the terminal output; the MCP server instead returns structured JSON with summary counts + `nextActions`.

**The 11 parity rules** (full catalog in `component-inventory.md`):

| Rule id | Enforces |
| --- | --- |
| `sot.project-json` | Canonical `.project.json` shape (keys, `repo_path`, agents, `ticket_provider`, `automation`); removes legacy `.plane.json` / `board_url` |
| `mise.config-root` | `mise.toml` present with the AGENTS-link enter hooks + `op inject` + `_.path` entries |
| `mise.versioning` | The managed `mise-versioning` task block + `.mise/scripts/versioning.sh` |
| `sot.agent-symlinks` | `CLAUDE.md`/`GEMINI.md` are symlinks to `AGENTS.md` |
| `secrets.env-op` | `.env.op` / `op inject` secrets wiring |
| `provenance.copier` | copier answers/provenance present |
| `bmad.scaffold` | BMAD (`bmm,bmb,cis`) installed for the full tool matrix |
| `bmad.version` | Installed BMAD version matches the `next` channel (cached npm dist-tags, <1h TTL) |
| `hermes.pm-scaffold` | The PM role scaffold under `agents/hermes/pm` is complete |
| `hermes.untracked-runtimes` | Hermes runtime submodules are untracked/gitignored, not committed |
| `systemd.sentinel` | Hermes `--user` gateway/consumer/heartbeat units are enabled + active |

The engine contains substantial **TOML surgery** for `mise.toml` (hook-block strip/reinsert, `_.path` merge, managed-block replacement) written to be **space-safe** (single-quoted `{{config_root}}`) and to use `[[hooks.enter]]` array-of-tables — mise mangles the `enter = [ … ]` string-array form. See the "Known gotchas" section.

### 4. Hermes agent provisioning (`src/commands/hermes/`)

The `hermes-agent` recipe chains seven commands (each mutates a shared `HermesAgentContext`):

1. `EnsureTemplateConfig` — bootstrap `~/.config/hermes-agent-template/config.toml` if missing
2. `PromptForAgentConfig` — TUI (or `--yes` defaults)
3. `RunCopierTemplate` — `copier copy` the Hermes template
4. `UntrackHermesRuntimes` — untrack the runtime submodule + gitignore it
5. `WireTelegram` — BotFather token capture (skippable)
6. `WireEmail` — Cloudflare Email Routing rule (opt-in via `--email`)
7. `PrintHermesSummary` — connection points + next commands

Heavy/irreversible steps (runtime GH repo, ticket-board creation, Bloodbank consumer, systemd `--user` units) are **off by default under `--local`** (and systemd is auto-skipped on macOS) so a non-technical operator can't accidentally create cloud resources. Fleet-wide operations belong to the `agent-fleet-operations` skill, not this repo.

### 5. Utilities (`src/utils/`)

- `version.ts` — `PJANGLER_VERSION` is **read from `package.json` at runtime** (walks up from the module dir so it works in `src/` dev and bundled `dist/`). This exists because of PJAN-2: a hardcoded literal made `pj --version` report `1.0.0` forever after bumps.
- `style.ts` — ANSI color/glyph helpers (`bold`, `cyan`, `dim`, `glyph`, `heading`, status colors) shared across CLI + parity output.

## The two interfaces

### CLI surface (`src/index.ts`)

`pjangler` / `pj` — commander program. Top-level commands:

| Command | Purpose |
| --- | --- |
| `init [name]` | Full project bootstrap (registry + CommonProject scaffold + `.project.json`). Inside an existing git repo → **sync mode** (audit + selective migrate). |
| `add <subsystem>` | Scaffold a subsystem/recipe into the cwd (`mise`, `docker`, `node`, `agent-hooks`, …) |
| `list` | List available subsystems |
| `project init/list/show/doctor` | Manage the central project registry |
| `recipe list/describe/run` | Recipe metadata + execution |
| `command|cmd list/describe/create` | Command metadata (`create` is a STORY-005 placeholder) |
| `audit [repo]` | Deterministic parity audit (exit 1 if not ok) |
| `migrate [rule-id] [repo]` | Idempotent migration; `--all`, or interactive rule selector TUI |
| `hermes-agent` / `hermes` | Provision the PM agent for the current repo |
| `config bootstrap` | Create the host `hermes-agent-template/config.toml` |
| `describe` | Describe the current project (placeholder) |

**Deprecated shims (still work, warn):** `pjangler init <subsystem>` → `pjangler add <subsystem>`; `pjangler project init` → `pjangler init`.

### MCP tool surface (`src/mcp-server.ts`)

`pjangler-mcp` over stdio. 11 tools (README lists only 4 — that list is **stale**):

`pjangler_list_capabilities`, `pjangler_list_parity_rules`, `pjangler_audit_project`, `pjangler_migrate_project`, `pjangler_bootstrap_33god_project`, `pjangler_project_init`, `pjangler_project_list`, `pjangler_project_show`, `pjangler_describe_recipe`, `pjangler_run_recipe`, `pjangler_deploy_hermes_agent`.

MCP defaults are **safety-first**: `dryRun` and `local` default to `true`, network/cloud actions require explicit opt-out. `runRecipeWithCapture()` redirects `console.log/error` to capture recipe output into structured `{ success, logs, errors }`. Every response embeds `parityGuidance()` pointing agents at the `@33god-projects` skill.

## End-to-end flows

**`pjangler init myproj --apply --provision-agent`**
1. `resolveProjectInitTarget` → decides create vs sync, resolves target dir/name.
2. `planProjectInit` → `ProjectInitPlan` of typed actions.
3. (interactive) `@clack/prompts` multiselect lets the user pick which actions + parity fixes to run.
4. `executeProjectInitPlan` → upsert registry, `spawnSync('copier', …)` the CommonProject scaffold, write `.project.json`.
5. If sync mode: `runAudit` + `runMigrationForRules` reconcile the repo to parity.

**`pjangler audit` / `pjangler migrate`**
`runAudit(repo)` → findings; user (or `--all`) selects fixable rules → `runMigrationForRules` applies idempotent migrations; report printed / returned as JSON.

**Agent via MCP:** `pjangler_list_capabilities` → `pjangler_audit_project` → `pjangler_migrate_project` (dry-run first) — the recommended workflow embedded in the tool metadata.

## Testing strategy

No unit-test framework; correctness is guarded by **four Node `.mjs` regression suites** run via `npm test`:

- `parity-migrate-regressions.mjs` (~18.5 KB) — the parity audit/migrate matrix (largest suite)
- `project-registry-regressions.mjs` (~17 KB) — registry validation, plan/apply, doctor
- `mcp-server-regressions.mjs` — MCP tool behavior
- `mcp-catalog-regressions.mjs` — MCP catalog/capabilities stay in sync

`npm run typecheck` (`tsc --noEmit`) is the type gate. See `development-guide.md`.

## Known gotchas & self-parity notes

- **Generated mise hooks must be space-safe.** mise runs each hook through `sh -c` after expanding `{{config_root}}`; an unquoted path containing a space word-splits and fails. The engine single-quotes every `config_root` reference and emits `[[hooks.enter]]` tables (the `enter = [ … ]` array-of-strings form gets mangled into one broken argv). Recent commits (`8fda33b`, `3e0e36d`, `b2e3565`) hardened this.
- **The repo's own `mise.toml` still uses the legacy `[hooks] enter = [ … ]` unquoted form** — i.e. pjangler does not yet apply its newest generated form to itself. Harmless here (no spaces in the path) but a self-parity gap.
- **`templates/commonproject` and `templates/hermes-agent` are git submodules** and are not checked out in every worktree (they are empty in this `bmad-init` worktree). `resolvePjanglerRoot()` locates the package root by walking up until it finds `package.json` + `templates/commonproject/copier.yml`.
- **README drift:** the README references `bun install` / `bun run` and lists only 4 MCP tools; the project is npm-only (esbuild) with 11 MCP tools. Trust `package.json` and the code.
- **`copier` is an external Python tool** and must be on `PATH` (`uv tool install copier`); a missing `copier` surfaces as an `ENOENT` error in `executeProjectInitPlan`, not a crash.
